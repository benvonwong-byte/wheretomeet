Group Meetups & Fair Places

Thyme & Place · extending A/B → up to 5 people · design v1
Three decisions you locked:
  • Fair = a blend — first cap the longest single trip (nobody stuck traveling way more than the rest), then among near-ties prefer the lowest total.
  • The dial becomes a Fairness ↔ Efficiency slider in group mode (one knob that tunes that blend).
  • 2-person stays exactly as it is today. Headcount picks the mode: 1 = near-me, 2 = the polished A/B experience, 3–5 = the new group model.

01What "fair" means for a group

At any candidate spot, each person has a travel time. Two numbers summarize the group: worst = max(tᵢ) (the person who travels longest) and mean = Σtᵢ / N (the average trip). The blend is a single cost the slider tunes:

# λ = 0 → strict fairness (minimax) · λ = 1 → efficient (least total)
cost(λ) = lerp(worst, mean, λ) = (1−λ)·worst + λ·mean
score = exp(−cost / 35) # higher = better, same shape as today's fairnessScore

The current 2-person formula (exp(−total/45)·exp(−evenness²)) is left untouched for N=2. The blend above is the N≥3 generalization — worst-trip is the fairness lever, mean is the efficiency lever.

03Scoring & ranking (group)

  • One travel-time field per person, keyed ${personId}:${mode}:${daypart} and cached independently — moving one pin only recomputes that person.
  • Per venue: sample each person's field → model times; refine the top spots with real street-routed times per person (per-person exactCache), exactly like today.
  • Rank by the blend score. Viability gate: only surface venues whose worst trip ≤ (best achievable worst) + ~12′ — so the list is genuinely fair spots, never "great for four, terrible for one."
  • Venue row shows the number that matters to a group — the longest trip — plus total, e.g. ⏱ 28′ longest · 78′ total. Per-person breakdown lives in the detail card.
  • Detail card: one color-coded row per person (their time + mode), plus a worst / total summary line.

04The map

People get five distinct, colorblind-reasonable colors (white-ringed bullets & markers):

P1 (you) green · P2 purple · P3 teal · P4 coral · P5 blue

Fair-zone glow (3+): the venue-anchored heat becomes a single warm-green intensity — brightest where the blend score is best, fading to nothing elsewhere. It literally shows the fair zone, and a single hue avoids the "what does this color mean with 5 people?" problem. Map key: "brighter = fairer for everyone." (If P1-green competes visually with the glow, the glow shifts toward gold — a fine-tune, not a fork.)

Same slider element as the A/B dial. In group mode it sets λ ∈ [0,1] (default ≈ 0.35, leaning fair); dragging re-scores venues and reshapes the glow live. In 2-person mode it stays the directional dial. One control, context-aware.

06UI — adding & removing people

  • The two fixed A/B blocks become a rendered list of person rows (address + mode pills + name-on-tap, as today), one per person.
  • ＋ Add a person button below the list while count < 5. Each person past the first has a small × to remove.
  • Person 1 is "you" and can't be removed (it's the near-me anchor). Removing back down to 2 → duo, to 1 → near-me — the modes flip automatically.
  • The A↔B swap button only makes sense for two people, so it hides at 3+.
  • Mobile: the person list lives in the existing plan-editor sheet; the current "Add a friend" invite becomes "Add people." Bottom sheet, chip strip, everything else unchanged.

07Data model & share links

The core refactor: replace state.A / state.B with state.people: Person[] (index 0 = you). people.length drives the mode. This is the biggest change — A/B are referenced across scoring, fields, markers, detail, and share — so it ships behavior-preserving first (see phasing).

Share URL generalizes to a repeated person param, e.g. p=lat,lng,mode,name per person, plus l= for the group λ. Old a=/b=/t= links keep working (parsed into people[0..1] + directional bias) so every link already shared stays valid.

08Performance (your machine crashes under load — taken seriously)

  • N ≤ 5 is a hard cap. Worst case is 5 transit Dijkstra fields on a pin move — but fields are cached per person, so moving one pin recomputes one field, not five.
  • Street-route draws on a venue tap go from 2 legs to N; the geometry cache already dedups, and it only fires on tap. Blend scoring over the grid is O(cells × N) — negligible.
  • No new servers, no background polling. Same footprint as duo, scaled by a small constant.

09Testing

  • groupScore: minimax at λ=0, mean at λ=1, monotonic in between; the worked example above becomes a fixture.
  • Viability gate keys on worst, not total.
  • Share round-trips N people and back-compat: an old a=/b=/t= link still loads as a 2-person directional plan.
  • Headcount transitions 1↔2↔3 flip modes/among slider/heat correctly.

10Phasing — two ships, each verifiable

1
Refactor to people[], behavior-preserving. No user-visible change: 1 and 2 people work identically to today, all 57 tests green. De-risks the big rename before any new behavior rides on it.
2
Add the group model. Blend scoring + fair-zone glow + fairness slider + add/remove-people UI + generalized share. This is where 3–5 people light up.

11Risks & open questions

The A/B → people[] rename is broad. Mitigated by shipping it behavior-preserving first (phase 1) with the full test suite as the safety net.
Glow hue vs. person-green. Resolvable in implementation (shift glow toward gold if it reads as "P1's zone"). Flagged, not blocking.
Venue row density. Showing "longest + total" keeps rows scannable; the full per-person list is one tap away. If that still feels thin, an optional expand-in-row is a cheap follow-up.
Open question: should the viability slack (~12′ on the worst trip) be fixed, or itself move a little with the slider (stricter fairness → tighter slack)? Leaning fixed for v1; easy to revisit.
