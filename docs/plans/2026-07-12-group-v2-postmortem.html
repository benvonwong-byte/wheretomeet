Group Meetups v1: what actually broke → v2 fork

Thyme & Place · post-mortem of the reverted 3–5 person feature · branch group-v2 · main untouched

01The smoking gun

One shadowed variable caused most of what you experienced. In the address-suggestion dropdown:
hits.forEach((h, i) => { // ← this i is the DROPDOWN ROW NUMBER…
  row.onmousedown = () => applyLocation(i, h, input); // …but applyLocation expects the PERSON index
});

So mouse-clicking the first suggestion in any person's box always moved Person A. Clicking the 2nd moved B. Clicking the 3rd suggestion in a 2-person plan threw a TypeError and silently died. The text landed in the box you typed in, but the pin, state, and URL mutated a different person — the exact "address bar and map de-synced" you reported. Keyboard-Enter used the correct index, which is why it slipped past my automated checks: they pressed Enter; you clicked.

03Why v1 kept bleeding — five structural causes

1 · Three sources of truth, patched by hand. state.people, the DOM inputs, and the map markers each held part of the truth. Every mutation path (type address, click suggestion, drag pin, add, remove, swap, near-me, mode click) had to remember to update all three. Each bug was one forgotten update. There were ~10 mutation paths × 3 surfaces = 30 chances to forget.
2 · Everything keyed by array index. people[], markers[], exactCache[] spliced in parallel; fieldCache keyed "2:transit"; async callbacks capturing i. Every remove shifted every index under everything in flight.
3 · The solo-mirror hack. "Near-me" was faked as two people where B secretly copies A. That one lie produced the phantom-B, the stale-marker, the uncommitted-text wipe, and the cache leaks at every solo↔group transition.
4 · Back-compat getters (state.A, nameA) kept old code half-alive. The swap bug is the poster child: an obsolete name-swap line still "worked" through the setters and silently un-did the real swap.
5 · Verification exercised the keyboard path, never the mouse path. Process failure, not code: the matrix must drive the app the way you do.

04The v2 architecture (branch group-v2)

Stable IDs, one source of truth, declarative render. Same product design you approved (blend fairness, λ slider, per-person colors); rebuilt internals.
  • Person = { id, pt, mode, name, label } — a monotonic id that never shifts; a single mode (the Set was dead generality); and the committed address label lives on the person, not in the DOM.
  • No solo-mirror. Near-me is simply people.length === 1. Person B does not exist until committed. The UI always shows a ghost slot — the next person's address box with live mode pills — so nothing is hidden; committing it creates the person.
  • ID-keyed everything. markers: Map<id, Marker>, exactCache: Map<id, …>, fieldCache["id:mode:daypart"]. Removing a person purges their id; nothing else moves. Swap becomes a pure array reorder — zero cache surgery (v1's swap needed 20 lines of cache renames; v2 needs none).
  • One renderPeople() rebuilds every row from state after any change — labels, pills, bullets, remove buttons all reflect state by construction. No per-slot patching, no compat getters, mouse and keyboard share one commit path that captures the person id, never a loop index.
  • Async safety: callbacks capture {id, pt} and re-resolve the person by id on completion; caches are id-keyed so a stale write can't land on the wrong person. Roster changes close the detail card and bump route/refine tokens.
  • Share links: ≤2 people use the exact current wire format (all existing links keep working). Extras ride in p=, parsed from the raw hash and decoded exactly once inside a try/catch — %, commas, and semicolons in names/addresses round-trip safely, and a malformed link can never blank the app.
  • Modes by headcount: 1 = near-me closeness glow · 2 = today's duo, pixel-identical (dial, advantage heat, swap, Δ table) · 3–5 = blend ranking, λ slider, per-person colored heat (each person's color, blended where even — the thing you asked for).

05Verification bar before it's called anything

  • The full matrix on desktop and mobile, exercising mouse clicks on suggestions (row 1, row 2, row 3+) for every person slot — the path that hid the smoking gun.
  • Roster torture: add→remove→add, near-me↔duo↔group round trips, swap with names+addresses set, drag-pin after typing, remove during in-flight routing.
  • Share round-trips incl. "%"/comma names, old-format links, and cross-checking pin positions against state after every step.
  • All unit tests + an independent adversarial review of the diff before the branch is offered for a test drive. Main and the live site stay untouched throughout.
